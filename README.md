# Detecting LLM-Generated Student Essays

**Joseph Souaiby & Huzaifah Wasim**

This repository contains our CS 7643 project on detecting whether a student essay was written by a human or generated by an LLM. The project focuses on two goals:

1. Build competitive essay-level detectors.
2. Measure how well those detectors transfer under distribution shift instead of only reporting in-domain benchmark accuracy.

At the current stage, the repo includes:
- TF-IDF baselines with logistic regression and linear SVM
- A GLTR-style likelihood baseline built from language-model token statistics
- Exploratory data analysis for the Kaggle and RAID datasets
- Out-of-domain evaluation on RAID
- A milestone report written in the ICLR-style LaTeX template

## Project Summary

We formulate the task as binary text classification:
- Input: raw essay text
- Output: probability that the essay is AI-generated

The main training source is the Kaggle **LLM Detect AI Generated Text** dataset. We then evaluate transfer to the **RAID** dataset to test robustness under a different distribution.

The current project direction is:
- Establish strong classical baselines
- Diagnose where they fail under domain shift
- Fine-tune transformer encoders such as BERT or RoBERTa as the main next-stage model

## Current Baselines

### 1. TF-IDF + Linear Models

We build sparse unigram/bigram TF-IDF features and train:
- Logistic Regression
- Linear SVM

These baselines perform extremely well in-domain on Kaggle, but they degrade sharply on RAID, which suggests they are picking up dataset-specific lexical or formatting artifacts.

### 2. GLTR-Style Likelihood Features

We also implement a likelihood-based baseline inspired by GLTR-style detection. Essays are scored with a causal language model, and we aggregate features such as:
- mean token log-probability
- surprisal statistics
- token rank statistics
- entropy statistics
- top-k token-rank shares

These features are fed into:
- Logistic Regression
- Linear SVM

This baseline is weaker than TF-IDF on Kaggle but generalizes better to RAID.

## Current Results Snapshot

Approximate milestone-stage results from `Baselines.ipynb`:

| Dataset | Method | Accuracy |
|---|---|---:|
| Kaggle | TF-IDF + Logistic Regression | 99.12% |
| Kaggle | TF-IDF + SVM | 99.74% |
| Kaggle | GLTR + Logistic Regression | 95.42% |
| Kaggle | GLTR + SVM | 95.64% |
| RAID | TF-IDF + Logistic Regression | 40.86% |
| RAID | TF-IDF + SVM | 40.90% |
| RAID | GLTR + Logistic Regression | 57.78% |
| RAID | GLTR + SVM | 55.20% |

The main takeaway is that high Kaggle accuracy alone is misleading. The GLTR-style baseline currently provides the better robustness reference point for the next transformer stage.

## Environment Setup

This project uses [`uv`](https://docs.astral.sh/uv/) for dependency and virtual environment management.

### 1. Install `uv`

See the official installation instructions:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

On Windows, install `uv` using the official installer or PowerShell instructions from Astral.

### 2. Clone the repository

```bash
git clone <repo-url>
cd DeepLearningProject
```

### 3. Create the environment and install dependencies

```bash
uv sync
```

This creates a local `.venv` and installs the packages declared in `pyproject.toml`.

### 4. Launch Jupyter

```bash
uv run jupyter notebook Baselines.ipynb
```

## GPU / PyTorch Notes

This repo is configured to use a CUDA-enabled PyTorch build from the official PyTorch wheel index.

If the environment is set up correctly, this should report a CUDA device:

```bash
uv run python -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'CPU')"
```

If `torch.cuda.is_available()` is `False`, your environment is probably using a CPU-only PyTorch build or the notebook kernel needs to be restarted.
